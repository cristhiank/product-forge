## Goal

Add validateWorker(workerId, opts?) to WorkerManager that checks actual work output (git commits, file scope, build result) instead of relying on worker self-reports.

## Problem
B-031.11 reported complete but created files in verticals-forms-api/ instead of verticals/pet_boarding/. Orchestrator had to manually detect scope leak. Workers can silently succeed at the wrong thing.

## Interface
validateWorker(workerId, opts?): ValidationResult
- opts.buildCommand: string — e.g. "dotnet build Kania.slnx -v minimal"
- opts.requiredPathPrefixes: string[] — files must be under these paths
- opts.forbiddenPathPrefixes: string[] — files must NOT be under these paths
- opts.requireCommits: boolean — default true, fail if no commits

## ValidationResult
- valid: boolean (overall pass/fail)
- hasCommits, commitCount, commitMessages: string[]
- filesChanged: string[]
- scopeViolations: string[] (files outside required or inside forbidden)
- buildPassed: boolean | null, buildOutput: string
- errors: string[]

## Logic
1. git log --oneline HEAD..worker/<id> for commits
2. git diff --name-only HEAD...worker/<id> for changed files
3. Check paths against prefix filters
4. Run buildCommand in worktree if provided
5. Return structured result

## Files
- src/workers.ts, src/sdk.ts, src/cli.ts, src/types.ts, SKILL.md, tests/workers.test.ts

## Acceptance Criteria
- Returns scopeViolations when files outside allowed prefixes
- Returns hasCommits: false when no commits on branch
- Runs build in worktree and captures pass/fail
- SDK and CLI expose the method
- Tests cover scope pass, scope violation, no commits, build pass, build fail
