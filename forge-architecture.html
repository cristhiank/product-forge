Core Concept

The Why Loop / How Loop Boundary

The fundamental architectural principle: the Coordinator controls intent (why), while Subagents handle implementation (how). The Coordinator never edits files, never runs builds, never writes tests.

⬡ The Why Loop Coordinator

Classify the user's intent → Route to the right mode → Dispatch via Mission Brief → Evaluate the result against the objective.

The coordinator sees the full picture: what was asked, what tier it is, which phase comes next, and whether the output matches the intent. It orchestrates but never implements.

⇄

⬢ The How Loop Subagents

Each subagent receives a Mission Brief with objective, constraints, and DONE WHEN criteria. It operates within a single mode, uses only its permitted tools, and returns structured artifacts.

Subagents are stateless and sandboxed — they can't see other phases, can't route themselves, and can't skip quality gates.

Complexity Classification

Five Tiers Shape Every Decision

Each tier determines which phases are required, tool budgets, verification depth, and how many plan steps are generated. The coordinator assigns a tier in the EXPLORE phase.
Tier Name Description Required Phases Plan Steps Verify Depth
T1 Trivial Quick answer, no file changes. Pure knowledge response. Answer inline 0 None
T2 Simple 1-2 files, well-known pattern, low ambiguity. Typos, config tweaks, simple additions. Explore Execute Verify 1–3 Spot-check
T3 Moderate 3-8 files, some design decisions needed, cross-module changes. Features with clear scope. Explore Assess ⚡ Design Plan Execute Verify 3–8 Standard
T4 Complex 8+ files, multiple modules, architectural decisions, risk of scope drift. New subsystems. All phases + Ideate 5–15 Thorough
T5 Ambiguous Unclear scope, multiple valid interpretations, cross-team impact. System redesigns, migrations. All phases + Experts Council 10–15+ Thorough + Council

🔄 Loop Mechanics

Agents execute tasks → Harness collects verify results, retries, scope drift, user satisfaction → Retrospective analyzes patterns → Proposes patches to mode files, templates, and gates → Next sessions run improved harness.

🧠 Memory Extraction

The MEMORY mode extracts durable knowledge from session trails, scoring facts on a 0–1 scale (threshold: 0.5). Cross-session correlation identifies patterns that no single session could see. Memories are deduped and stored for future context.

forge-explore
Codebase investigation with fact confidence levels. Read-only tools (grep, glob, view, git). Budget allocation by complexity. Outputs tier classification.
forge-ideate
Generate 2-3 meaningfully different approaches. Must differ in 2+ dimensions. Always includes contrarian option. Web search only — no codebase tools.
forge-assess

Atomic execution plan with dependencies and DONE WHEN criteria. Micro-plan (T3: 3-8 steps) or full-plan (T4-T5: 5-15 steps).

forge-retrospective
Failure analysis and harness improvement. Single root cause classification. Max patches per tier (T2→2, T3→3, T4-T5→5). Never speculative.
forge-gc
Codebase health scanning. Deterministic + LLM analysis. Debt, dead code, stale docs, architectural entropy. 2-7 backlog items per scan.
forge-memory
Extract durable knowledge from trails. User-request only. Scoring 0.5+ threshold, <200 chars per memory. Cross-session correlation.
forge-product
Manage .product/ repo: feature specs, discovery docs, experiments, lifecycle transitions. CLI-driven via product-hub. SUCCESs self-review.
skill-creator
Bootstrap new skills with proper SKILL.md structure, references directory, and script scaffolding.
mcp-builder
Build MCP (Model Context Protocol) servers for tool integration. Python (FastMCP) or Node/TypeScript patterns.
code-indexer
Index .NET/C# solutions, search symbols via full-text documentation search, generate structural overviews.

Quality Gates & Constraints

Guardrails at Every Phase

The system enforces constraints that prevent common agentic failure modes: scope drift, hallucinated implementations, rubber-stamp verification, and speculative fixes.
🛡️ Design Guard

For T3+ tasks, ASSESS and DESIGN are mandatory before PLAN/EXECUTE — even if the user says "just implement it". The coordinator enforces this automatically.

🔒 Coordinator Sandbox

The coordinator never edits files, runs builds, or writes tests. All mutations go through subagent dispatch. This prevents mixing intent classification with implementation.

🔍 Verify 2-Pass Limit

Verification gets at most 2 passes. After 2 revision_required verdicts, the system escalates to the user rather than looping. Prevents infinite correction cycles.

📏 Scope Check

PLAN phase begins by comparing plan scope against the approved design. If scope drifts beyond the design boundary, the plan is rejected before execution begins.

🚫 No Rubber-Stamping

VERIFY mode has explicit anti-patterns: no "LGTM" without evidence, no skipping checklist items, no approving code you haven't inspected. Every verdict needs specific file references.

📊 Tier-Calibrated Budgets

Each tier gets a different tool budget, plan step count, and verification depth. T1 gets zero tools; T5 gets full multi-model council review. No one-size-fits-all.

🧪 Contract-Driven TDD

EXECUTE follows "Code Little → Test Little" cycles. Write 10-15 lines, then verify. Contracts from DESIGN become test assertions. Tests are written before implementation.

🔄 Retrospective Patches

RETROSPECTIVE proposes harness improvements, not code fixes. Patches are scoped by tier (T2→max 2, T5→max 5) and require user approval before applying. Never speculative.
